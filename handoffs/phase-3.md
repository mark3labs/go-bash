## What this phase delivered

- **`fs/fs.go`** — `FileSystem` interface (embeds `iofs.FS`, `iofs.StatFS`, `iofs.ReadDirFS`; adds OpenFile/Create/Mkdir/MkdirAll/Remove/RemoveAll/Rename/Symlink/Link/Readlink/Lstat/Realpath/Chmod/Chtimes/ReadFile/WriteFile/AppendFile/AllPaths). `File` interface (io/fs.File + io.Writer + io.Seeker + Truncate). `FileInit` struct with Content / Lazy / Mode / Dir / Symlink fields. Cross-cutting `PathError` helper and the sentinel errors `ErrReadOnly`, `ErrSymlinkLoop`, `ErrNotDirectory`, `ErrIsDirectory`.
- **`fs/path.go`** — `Validate` (mandatory null-byte + empty-path rejection), `Clean`, `Join`, `Resolve`, `Dirname`, `Basename`, `IsWithinRoot`, `SplitParts`, `MaxSymlinkDepth = 40`. All FS implementations call `Validate` at every public entry point — null-byte injection is impossible to express through the API.
- **`fs/fstest/fstest.go`** — shared contract suite. Each implementation's `_test.go` calls `fstest.Run(t, Suite{...})` with capability flags (`SupportsSymlnk`, `SupportsHardLn`, `SupportsChmod`, `SupportsChtime`). Tests: null-byte rejected, write+read, OpenFile read/write, Truncate, AppendFile, Stat regular, Mkdir+ReadDir, MkdirAll recursive, Remove, RemoveAll, Rename, HardLink, Symlink+Readlink+Lstat+Stat-through-link, Realpath, Chmod, Chtimes, fs.WalkDir, ReadFile-not-exist, ReadFile-of-directory.
- **`fs/memfs`** — in-memory FS with regular files, directories, hard links via shared `*fileContent`, symlinks resolved at every public entry, lazy file providers, sync.RWMutex protection. `Seed(map[string]FileInit)` honors `BashOptions.Files` at construction time, including lazy providers that fire only on first read.
- **`fs/realfs`** — shared host-disk security helpers: `ResolveAndValidate` enforces sandbox containment and rejects intermediate symlinks (when `AllowSymlinks=false`), `OpenFileNoFollow` adds `O_NOFOLLOW` on Unix (build-tagged), `SanitizeError` strips the sandbox root from `*os.PathError`. Build files split unix/non-unix to keep the runtime portable.
- **`fs/rwfs`** — direct read-write FS under a sandbox `Root`. Symlinks stored as host-absolute paths and inverse-mapped on `Readlink` so virtual paths round-trip through Symlink/Readlink. Path traversal attempts (`/../etc/passwd`) are rejected via `realfs.ResolveAndValidate`.
- **`fs/overlayfs`** — overlay-on-disk FS. Reads served from the overlay if present, else from the host subtree; writes always land in the in-memory overlay; deletes drop a tombstone; listing unions the two and removes tombstoned children. `ReadOnly: true` returns `ErrReadOnly` on every mutating op. `AllowSymlinks` defaults to false (host symlinks hidden) for security parity with just-bash.
- **`fs/mountfs`** — composition FS. Longest-prefix mount resolution; cross-mount Rename/Link fall back to copy+delete; base directory listings synthesize entries for any mount that sits directly under the queried path (so `ls /` shows mount points even when the base FS doesn't have them).
- **`bash.go` — VFS wiring:**
  - `Bash.fs` field added; populated at `New()` from `BashOptions.FS` (when set) or a fresh `memfs.New()`.
  - `BashOptions.Files` seeded into whatever FS we end up with — via `*memfs.FS.Seed` (fast path, supports lazy providers) or via the generic `FileSystem` API for any other backend. Null-byte paths cause `New()` to return an error.
  - Default cwd: `/home/user` when neither Cwd nor Files is set, else `/` — both auto-`MkdirAll`'d into the FS so mvdan/sh's initial pwd stat succeeds.
  - `Bash.FS()` accessor returns the bound FS.
  - mvdan/sh's `OpenHandler`, `StatHandler`, `ReadDirHandler2` all replaced with VFS-backed implementations that resolve the requested path against the current handler `Dir` (panic-safe — mvdan/sh's `Runner.stat` calls the handler with a bare ctx without a HandlerContext, so the extraction is recover-guarded).
  - **mvdan/sh `interp.Dir()` workaround:** `interp.Dir(path)` runs an `os.Stat(path)` against the host disk at runner-init, which fails for virtual paths like `/home/user`. We construct the runner without `interp.Dir` and then set `runner.Dir = cwd` directly. This bypasses the host-side stat and our VFS supplies the right answer the moment the runner actually executes anything.
- **Phase 2 tests unchanged:** `TestMaxLoopIterations`, `TestMaxLoopIterationsForLoop`, `TestMaxOutputSize`, `TestMaxOutputSizeCountsStderr`, `TestMaxCallDepth`, `TestMaxCommandCount`, `TestLimitsDoNotImpactNormalScripts`, `TestLoopSentinelInvisible` all still green.
- **Lint-cleanup:** every `Close()` call on a deferred / error-path file handle in memfs / rwfs / overlayfs now either uses `defer func(){ _ = f.Close() }()` or `_ = h.Close()`, clearing the 8 errcheck warnings golangci-lint surfaced after the initial commit.

## Decisions & gotchas discovered

- **`interp.Dir()` does a host-side `os.Stat` at runner construction.** This is invisible from the godoc but unambiguous from the source — `interp/api.go:335` calls `os.Stat(path)` directly, bypassing our `StatHandler`. For virtual paths like `/home/user` the call fails with "could not stat: no such file or directory" because the host doesn't have that path. **Workaround:** drop the `interp.Dir(cwd)` runner option and instead set `runner.Dir = cwd` immediately after `interp.New()` returns. The runner accepts the assignment without re-stating, and our `StatHandler` answers the first stat the runner actually issues (e.g. during `pwd` or a redirect). Documented inline. The same trap will trip Phase 5's interp bridge if it tries to use `interp.Dir` for `ExecOptions.Cwd`.
- **`interp.HandlerCtx(ctx)` panics if no HandlerContext is in ctx.** mvdan/sh's `Runner.stat` / `Runner.lstat` pass the raw ctx — without a HandlerContext value — straight to the `StatHandler`. So a naive `interp.HandlerCtx(ctx).Dir` blows up on the first `[ -f /file ]` test. **Workaround:** `handlerDir(ctx)` recovers the panic and falls back to empty (which `gbfs.Resolve` treats as "treat path as absolute" for absolute inputs, which all our test cases are). If mvdan/sh ever exposes a non-panicking variant we should switch. Recorded in the package comment.
- **mvdan/sh's `OpenFile` consumer treats our `File` as `io.ReadWriteCloser`.** That's exactly what our handler signature returns. memfs's `*fileHandle` satisfies it; rwfs/overlay wrap `*os.File` in `osHandle` to keep the surface clean and to leave room for instrumentation later.
- **Symlink targets in rwfs require namespace translation.** The contract test calls `Symlink("/target.txt", "/link.txt")` and expects `Readlink("/link.txt") == "/target.txt"`. But naively storing `/target.txt` on disk makes the target unreachable (no host file at that path). **Fix:** rwfs translates virtual-absolute targets into host-absolute paths under `Root` at Symlink time, then inverse-translates on Readlink (strip the `Root` prefix back to a virtual path). Relative targets pass through verbatim. A side effect: rwfs symlinks are bound to a specific host `Root` directory and won't survive a root move — acceptable for sandbox use.
- **Memfs hard-link shared content is preserved across `WriteFile(..., O_TRUNC)`.** `OpenFile` with `O_TRUNC` does `n.content.data = n.content.data[:0]` rather than replacing the `*fileContent` pointer, so a write through link A is observable at link B. `TestHardLinkSharesContent` covers this.
- **OverlayFs path containment lives in `realfs.ResolveAndValidate`.** Any attempt to escape with `/../../etc/passwd` resolves to `<root>/etc/passwd` after `filepath.Clean`, fails the `pathWithin(root, real)` check, and returns `iofs.ErrNotExist`. The same helper rejects intermediate-symlink poisoning when `AllowSymlinks=false`.
- **Default Cwd auto-create vs. caller-supplied FS.** When the user passes a custom `FS`, we still `MkdirAll(cwd, 0o755)` — that's a write against their FS. We swallow any error since a deliberately read-only FS would refuse, and the user is expected to have pre-created the cwd themselves. Worth revisiting once Phase 5 wires `ExecOptions.Cwd` (the per-call cwd may want different semantics).
- **`fs.FileSystem` matches `iofs.ReadFileFS` by virtue of having `ReadFile`.** This means `iofs.WalkDir` works out of the box against any of our implementations — the contract `Walk` subtest exercises that.
- **`mountfs` synthetic directory entries.** When `ReadDir("/")` runs against the base FS and the base doesn't have a `/mnt` entry, we still need to surface `mnt` so a script's `ls /` shows mount points. `mountsUnder` walks the mount table for entries whose path starts with the queried directory and synthesizes `iofs.DirEntry`s. This makes the mountfs feel like a single namespace.
- **Lazy file providers live only in memfs.** The generic `seedFiles` path in `bash.go` rejects lazy entries with a non-memfs FS — there's no portable way to defer materialization through the public `FileSystem` interface. Acceptable since Phase 3 only requires it for the default in-memory backend.
- **Phase 2 instrumentation is unaffected.** AST loop instrumentation, the `runtime.Callers` call-depth walk, and the MaxOutputSize cancel-and-stash trick all kept their wiring. The new handlers are extra `interp.RunnerOption`s composed alongside the existing `CallHandler`.
- **Lint quirks.** Staticcheck (via the Makefile) initially warned `S1016` on `mountfs.mountInfo` vs `mountEntry` and `S1017` on a manual `strings.HasPrefix` strip in `fs/path.go`. Both fixed inline. After the first finalize, golangci-lint (errcheck) flagged 8 unchecked `Close()` returns in memfs/rwfs/overlayfs ReadFile/WriteFile/AppendFile helpers — all converted to `defer func(){ _ = f.Close() }()` for the read path and `_ = h.Close()` for the error-path discards.

## Open follow-ups (non-blocking)

- **Phase 5 should not use `interp.Dir`.** Set `runner.Dir = cwd` directly after `interp.New()` to preserve the VFS-only stat path. The current comment in `bash.go` is a breadcrumb.
- **Tighten the `handlerDir` recover.** If mvdan/sh ever exposes a non-panicking `HandlerCtxOk(ctx)` we should switch; failing that, vendor a wrapper. The recover is cheap but it does hide programmer errors if we ever build a handler that genuinely needs `HandlerCtx`.
- **`MaxFileDescriptors` enforcement** — promised by Phase 2's open-follow-ups list. The FS layer now exposes `File` close hooks so a counter is trivial to add. Punt to Phase 5 with the interp bridge.
- **OverlayFs ReadOnly + AllowSymlinks combos.** Today `ReadOnly:true` blocks every mutating op but reads still copy-up on chmod via the OpenFile O_RDWR path. That's not reachable from outside (`writable()` short-circuits) but worth verifying when Phase 10's `chmod` builtin lands.
- **rwfs symlinks store host-absolute targets.** If the sandbox root moves the symlinks break. Replace with relative-to-link targets when we have time, or document the constraint.
- **`fstest` could grow O_EXCL, FIFO, device-file (skip), and per-impl symlink-loop tests.** Today symlink loops are exercised only through the resolver depth bound (40); a dedicated loop fixture would tighten coverage.
- **Coverage of `fs/realfs` is 0%** because all its calls go through rwfs/overlay and the tests target those. A focused unit test on `ResolveAndValidate` (especially the intermediate-symlink reject path on unix) would lift this.
- **`mountfs` synthetic-entry path** is reached only when the base FS has the prefix dir. If the base errors on `ReadDir("/")` we fall through to `synthesizeMountEntries`, which returns the mount points but loses any other ErrNotExist signaling — fine for memfs base, may need refining for read-only base FSes.
