## Decisions & gotchas discovered

- **`NewSecureFetch(nil)` returns a DENY-ALL Doer.** SPEC ┬¦9 wording is "defaults to DENY-ALL when `BashOptions.Network` is nil"; the Doer constructor handles nil so a hard-coded `network.NewSecureFetch(nil)` is the safe "lock everything down" sentinel. We don't currently exercise this from `bash.go` (we leave `b.fetch` nil when both options are nil) but a Phase 10 builtin that wants a sentinel Doer can grab one without ceremony.
- **`b.fetch` stays nil when both `Fetch` and `Network` are nil.** This is a deliberate split: the SPEC ┬¦9.3 "defaults to DENY-ALL" applies inside `NewSecureFetch`, but `command.Context.Fetch == nil` is the cleaner signal for downstream code that wants to print `network is disabled` rather than a generic `NetworkAccessDeniedError`. Phase 10's `curl` is expected to take the nil-check branch.
- **`%2f` / `%5c` rejection is on the raw input, not after `url.Parse`.** `url.Parse` leaves percent-encoded reserved chars intact in `RawPath` but not necessarily in `Path`; rather than reason about which field a future code path will read, we reject the literal substring (case-insensitive) anywhere in the entry URL. This is a strict superset of TS behavior (TS only checks the path) and intentionally rejects the encoded form in userinfo / query too. If a comparison fixture later needs a `?q=%2f` query, the rejection scope is the one-line fix in `containsEncodedSlash`.
- **Default-port stripping covers HTTP/HTTPS plus WS/WSS.** SPEC ┬¦9.2 says "default ports 80/443 stripped"; `isDefaultPort` extends that to `ws`/`wss` so future websocket-aware commands (Phase 10's `curl` does not need them, but a future `wscat` would) get the same normalization. Non-listed schemes preserve their port verbatim.
- **`headerTransform` calls `req.Header.Set(k, v)` WITHOUT pre-canonicalizing.** `http.Header.Set` already canonicalizes the key internally ŌĆö calling `CanonicalHeaderKey` first triggers staticcheck **S1035**. The Phase 9 first pass had the redundant call; this is now documented inline so a future contributor doesn't "fix" it the wrong way.
- **`http.Client` wraps every transport error in `*url.Error`.** `Doer.Do` calls `unwrapTypedNetErr` on the wrapped error to surface the typed error directly. This matters for `errors.As(err, &NetworkAccessDeniedError{})` at the caller ŌĆö the Phase 9 acceptance test `TestPhase9TypedErrorsRoutable` would fail without the unwrap, because `*url.Error.Err` is what `errors.As` would have to traverse, and the stub Doer path doesn't go through `*url.Error` but the real `SecureFetch` does. Tests on both sides confirm the unwrap.
- **`MaxRedirects=0` is reserved for "use the default".** Negative ŌćÆ disable. Positive ŌćÆ the cap. This matches the TS API but diverges from common Go idiom (where `0` usually means "disable"); we picked the TS reading because it's the parity-driving constraint. `MaxResponseSize` and `Timeout` follow the same three-way scheme (zero=default, negative=disable, positive=cap).
- **`DefaultAllowedMethods` is `[]string{"GET","HEAD"}` ŌĆö case-insensitive match via `strings.EqualFold`.** SPEC ┬¦9.1 just says "default" without a case rule; we picked case-insensitive because real HTTP servers accept any case and the alternative is forcing tests to upper-case everything. `MethodNotAllowedError.Method` preserves the caller's casing for diagnostic clarity.
- **`isPrivateIP` includes the full Go-stdlib `IsPrivate` set.** That includes IPv4 RFC1918 (10/8, 172.16/12, 192.168/16), IPv4 carrier-grade NAT (100.64/10), and IPv6 ULA (fc00::/7). Plus loopback (`IsLoopback`), link-local (unicast + multicast), regular multicast, and the unspecified address. SPEC ┬¦9.3 lists "RFC1918, ::1, fe80::/10, etc." ŌĆö the "etc." is broader than the bullet list, and we lean strict because a Phase 9 false-negative is a sandbox escape.
- **Tests use `httptest.Server` exclusively for the happy-path Doer plumbing.** `httptest.NewServer` binds to `127.0.0.1` ŌĆö combining that with `DenyPrivateRanges` would deadlock the test on the dial. The DenyPrivateRanges path is exclusively exercised via injected `DNSResolve` against a *non-existent* hostname, so the dialer never actually attempts to connect when every address is filtered out. `GOBASH_TEST_NO_NETWORK=1` is set by `make ci` and observed.
- **`Bash.fetch` is private; there is no `Bash.Fetch()` accessor.** This differs from Phase 8's `Bash.Registry()`. The Doer is per-Exec wiring; if a host wants to inspect it, the right tool is a `CustomCommand` that captures `c.Fetch`. Phase 9's wiring tests follow exactly this pattern. If SPEC ┬¦1.2 later wants an accessor, it's a one-line addition.
- **The `interp/Config.Fetch` field is consumed inside `registryDispatchMiddleware`, not `BuildRunner`.** This keeps the closure local to the dispatch path: a future Phase 11 sub-shell `Context.Exec` will need to re-bind the same Doer to a nested runner without `BuildRunner` knowing about it. Threading the Doer through the middleware constructor (not the handler-context lookup) makes that future swap easier.
- **`make ci` does NOT invoke `golangci-lint`; only staticcheck.** A separate `golangci-lint run ./...` pass was needed to catch errcheck. The Phase 9 first pass dropped 8 such diagnostics; commit 1aee8c3 fixed them all by assigning unchecked Write returns to `_, _ =`. The repo's `Makefile` lint target reads "staticcheck + custom lints (no-os-exec, no-net-default ŌĆö see SPEC ┬¦21.1)" so the custom-lint piece is still aspirational. Phase 21 hardening should fold golangci-lint into `make ci` either directly or via the custom-lint layer.

## Open follow-ups (non-blocking)

- **Phase 10 `curl` is the first real consumer of `c.Fetch`.** Today the field is exercised only by the `fetch-probe` test command. The Phase 10 kickoff prompt should remind the implementer that `c.Fetch == nil` means "network disabled, error out" ŌĆö not "fall back to host network".
- **`network.DefaultAllowedMethods` is exported and mutable (it's a slice).** A host that does `network.DefaultAllowedMethods[0] = "POST"` mutates the package default. The Phase 9 first pass copies it into `Config.withDefaults`, but the package-level slice is still racy if mutated. Phase 21 hardening should freeze it.
- **`Response.StatusText` parsing is naive.** It strips everything before the first space in `httpResp.Status` ŌĆö works for "200 OK" ŌåÆ "OK", "404 Not Found" ŌåÆ "Not Found", but if the underlying transport ever yields a `Status` without a space the index math would slice the whole string. Real-world `*http.Response.Status` is always `"NNN msg"`, so this is a future-proofing nit. Phase 10's `curl -w` will be the first reader.
- **No `DangerouslyAllowFullAccess + DenyPrivateRanges` matrix test for the redirect path.** The combo IS exercised for the initial request (`TestSecureFetch_DangerouslyAllowFullAccess`) but not for a cross-origin redirect under DenyPrivateRanges. Low risk because the dialContext path doesn't care which hop triggered it; flagging for Phase 10's curl test suite to cover.
- **`network/securefetch.go::resolve` has 17.6% coverage.** That's the production fallback to `net.DefaultResolver.LookupIP`. Untested by design (real DNS in CI is forbidden by `GOBASH_TEST_NO_NETWORK=1`). The DenyPrivateRanges path through `DNSResolve` (test injection) IS at 100%. If we want production coverage, Phase 19's comparison harness can mock `net.DefaultResolver`.
- **`MaxResponseSize` cap applies AFTER receiving the body, not via Content-Length pre-check.** SPEC ┬¦9.3 specifies `io.LimitReader(body, MaxResponseSize+1)`, which is what we do. A future enhancement could short-circuit on Content-Length > cap to avoid the wasted download. Non-blocking; documenting because a future curl `-w %{size_download}` may want to surface this.
- **No `HEAD`-with-body behavior test.** A HEAD request with the default cap might still receive a body from a non-compliant server; we'd cap it normally. Phase 10's `curl -I` should add this regression test.
- **`make ci` does not currently include `golangci-lint`.** Phase 21 hardening should add it; until then, run `golangci-lint run ./...` manually before handoff.
