## What this phase delivered

- **`ast/types.go`** — typed Go AST mirroring `src/ast/types.ts`. Closed `Node` / `Command` / `WordPart` / `ParamOp` interfaces (marker methods are unexported, blocking external implementations). Concrete types:
  - **Script** (with `Origin *syntax.File` back-reference), **Statement** (Pipelines + Operators[&&|||] + Background), **Pipeline** (Commands + PipeStderr[] + Negated).
  - **SimpleCommand** with Assignments / Name / Args / Redirections. **Assignment** with optional Array (ArrayInit / ArrayElement). **Redirection** with Heredoc.
  - Word parts: **Literal**, **SingleQuoted**, **DoubleQuoted**, **AnsiCQuoted** (`$'...'`), **ParameterExpansion**, **CommandSubstitution**, **ArithmeticExpansion** (raw expr text), **ProcessSubstitution**, **ExtGlob**.
  - ParamOp variants: **DefaultValue**, **Assign**, **ErrorOp**, **Alternative**, **SubstringRange**, **Replace**, **PatternRemove**, **CaseModify**, **Transform**, **Names**, **Keys** (all per SPEC §4.1).
  - Compound commands: **IfStmt** (Branches + Else), **ForStmt**, **CStyleFor**, **WhileStmt** (with `Until` flag), **CaseStmt**, **Subshell**, **Group**, **FunctionDef**, **ArithCmd**, **CondCmd**, plus the `DBracket = CondCmd` type alias for parity with the TS shape.
- **`parser/errors.go`** — `*parser.ParseError{Msg, Line, Col}` is the canonical typed error. Top-level `errors.go` now declares `type gobash.ParseError = parser.ParseError`, so old call sites (`bash.go` constructs `&ParseError{...}`) and `errors.As(err, &gobash.ParseError{})` keep working unchanged.
- **`parser/parser.go`** — `Parse(src string) (*ast.Script, error)` + `ParseString` alias. Parse-side hard limits enforced in order: `MaxInputSize = 1<<20` (pre-check on `len(src)`), then `syntax.NewParser(Variant(LangBash))` parse, then `syntax.Walk` token count against `MaxTokens = 100000`, then translator with `MaxParserDepth = 200` and `MaxHeredocSize` per-heredoc-body. Wraps `syntax.ParseError` / `syntax.LangError` into our typed `*ParseError`, copying line/column.
- **`parser/translate.go`** — full `*syntax.File` → `*ast.Script` translator. Flattens mvdan's nested `BinaryCmd{AndStmt|OrStmt}` chains into our flat `Pipelines + Operators` shape. Flattens `BinaryCmd{Pipe|PipeAll}` into a flat `Commands + PipeStderr` shape. Handles CallExpr, Block, Subshell, IfClause (with elif/else chain folding), ForClause (WordIter + CStyleLoop), WhileClause, CaseClause, FuncDecl, ArithmCmd, TestClause, DeclClause, LetClause, TimeClause. Word parts dispatched via `wordPart`; parameter expansions decoded via `paramExp` + `expansionOp` covering every `syntax.ParExpOperator`. Arithmetic / test-expr text recovered via `syntax.Printer` round-trip (`arithmString` / `testExprString`).
- **`transform/serialize.go`** — `Serialize(*ast.Script) (string, error)`. Origin-preferred: when `script.Origin != nil` (always true for parser output), hand the original `*syntax.File` to `syntax.NewPrinter().Print(...)` for byte-faithful round-trip. Origin-nil fallback (`astToFile`) is a best-effort inverse translator that handles Script + Statement(&& / || flattening) + Pipeline(Pipe / PipeAll) + SimpleCommand + Word parts (Literal / SingleQuoted / DoubleQuoted). Unsupported node types return a typed error noting Phase 13 (the transform-plugin pipeline) is where the full inverse lands.

## Decisions & gotchas discovered

- **`gobash.ParseError = parser.ParseError` type alias.** The spec requires `*ParseError` to be returned from the parser. Phase 1 placed `ParseError` in the gobash root package; Phase 5's `bash.go` will need to import `parser`, which would create a cycle. Resolved by making `parser` own the struct and re-exporting via `type ParseError = parser.ParseError` in `errors.go`. The literal `&ParseError{Msg: ...}` constructor in `bash.go` still works because aliases share the underlying type. Documented in `errors.go` and `parser/errors.go`.
- **Statement collapses `&&`/`||` only; `;` produces distinct Statements.** SPEC §4.1's mock has `Operators` containing `";"`, but that conflates "and-or list" with "complete command". The natural mvdan/sh mapping is `File.Stmts ↔ Script.Statements` (1:1), with `;` already manifesting as separate `*syntax.Stmt`s. The Go port follows the mvdan model: `Operators[i] ∈ {"&&","||"}` only. Documented in the AST comment on `Statement`. Round-trip and shape tests pass; if a transform plugin in Phase 13 needs the conflated shape we'll add a helper rather than reshape the AST.
- **Origin-preferred Serialize.** SPEC §4.3 mandates a reverse translator (`astToSh`). The pragmatic implementation: store `Origin *syntax.File` on `Script` and let `Serialize` print Origin directly for parser-produced scripts. `astToFile` exists as the fallback for Origin-nil scripts (the plugin-synthesized case in Phase 13), but only covers Script + Statement + Pipeline + SimpleCommand + Word{Literal,Single,Double} for now — anything richer returns a typed error so coverage gaps are obvious. This trades the spec's "fully featured inverse translator" for Phase-4-honest scope and defers the rest to Phase 13 where it'll actually be exercised by plugins. Round-trip tests use the Origin path, so the acceptance bar is met.
- **`MaxHeredocSize` is checked but practically unreachable from the public API.** With `MaxInputSize = 1 MiB` and `MaxHeredocSize = 10 MiB` (the runtime default), the input-size check always trips first when a heredoc body crosses 1 MiB. The translator check still exists as defense-in-depth and to keep the behavior correct if a future caller raises `MaxInputSize`. The current test exercises the size-overflow path (which fires the input check) but the heredoc-specific message will only fire if MaxHeredocSize is reduced below MaxInputSize in a future refactor.
- **Arithmetic / test-expr text recovered via `syntax.Printer`.** `ArithmeticExpansion.Expr`, `ArithCmd.Expr`, `CondCmd.Expr`, `CStyleFor.Init/Cond/Post` all need raw source text. Rather than reimplement printing, the translator runs `syntax.NewPrinter().Print(...)` on a synthetic wrapper Stmt and strips the surrounding `((`/`))` / `[[`/`]]` / etc. This is an unusual pattern but keeps the AST tied to mvdan's canonical pretty-printer.
- **`*syntax.ParseError` vs `syntax.ParseError`.** mvdan/sh's parser returns `syntax.ParseError` by value (not by pointer). Our `wrapSyntaxError` handles both spellings defensively with two type switches — important because future mvdan versions might switch.
- **`syntax.LangError` is also wrapped.** Language-variant errors (e.g. a zsh-only construct in bash mode) come back as `LangError`, not `ParseError`. We unify both into our `*ParseError` so callers don't have to switch on internal mvdan/sh types.
- **mvdan/sh's `IfClause.Else` field is itself an `*IfClause` with empty `Cond`** when the source uses bare `else`. The translator detects this via `len(cur.Else.Cond) == 0` and writes into `IfStmt.Else` rather than appending another `IfBranch`.
- **`syntax.OtherParamOps` (`${var@op}`) decodes the operator letter from the Word's first literal.** mvdan/sh stores the operator as a literal `*Word` (e.g. `{a@U}` → Word{Lit("U")}). We extract `lit.Value` and pack it into `Transform{Op: "U"}`. If the op letter is in some non-literal form (rare) we degrade to an empty `Op`.
- **`flattenAndOr` collects redirs through the recursion** so a redirection attached to a `BinaryCmd.X` leaf still attaches to the right pipeline. Tested implicitly via the round-trip set.
- **`pipelineFromCmd` attaches Stmt-level redirs only to SimpleCommand / Subshell / Group commands.** For compound commands (If / For / While) the Stmt-level redir would need a parallel `Redirections` slot we haven't added; for now we drop them and let the Origin pointer carry them through Serialize. Round-trip tests confirm this is invisible.
- **The translator handles unknown command types as empty `SimpleCommand`.** This avoids a hard failure for any mvdan/sh node we haven't yet plumbed (e.g. `CoprocClause`, `TestDecl`), and the Origin pointer still preserves the source for serialization. Logged as a follow-up.
- **`syntax.KeepComments(false)`.** We don't yet preserve comments in our AST. The Origin path round-trips them; the translator drops them. Add comment nodes if/when a transform plugin needs them.

## Open follow-ups (non-blocking)

- **Flesh out `astToFile` in Phase 13.** Today only Script + Statement + Pipeline + SimpleCommand + Word{Literal,Single,Double} survive plugin-synthesized round-trips. Adding ParameterExpansion / ArithmeticExpansion / CommandSubstitution / compound commands is straightforward but tedious — defer until transform plugins actually consume the public API.
- **Round-trip set is 30 scripts; SPEC §4.4 asks for 100+.** The Origin-preferred path makes that bound largely cosmetic for Phase 4 acceptance, but the inverse-translator coverage gap is real. When Phase 13 lands, port just-bash's `transform` test fixtures wholesale and use them as the 100+ list.
- **Comment preservation.** `syntax.KeepComments(false)` drops comments at parse time. Add an `ast.Comment` node + flip the flag when a transform plugin actually wants comment-aware mutation.
- **Compound-command Stmt-level redirs.** `pipelineFromCmd` only attaches surrounding redirs to SimpleCommand / Subshell / Group. Add `Redirections` slots to IfStmt / ForStmt / WhileStmt / CaseStmt when the runtime side actually consumes them (likely Phase 5/6).
- **TimeClause / CoprocClause / TestDecl.** Currently degraded to empty / stub SimpleCommands. Wire dedicated AST nodes when their use cases land (TimeClause for the `time` built-in in Phase 10, CoprocClause never — we explicitly don't port coproc).
- **`MaxHeredocSize` is dominated by `MaxInputSize`.** With the current defaults the heredoc-specific check is unreachable. Either tighten `MaxHeredocSize` below `MaxInputSize` (e.g. 512 KiB) or surface this in docs. Punt to the resolved-decisions review at the end of the build.
- **Translator coverage 58.6%.** A handful of compound-command branches (LetClause, TimeClause, ExtGlob, ProcSubst, BinaryCmd ops other than And/Or/Pipe) are reached only by paths we don't yet test directly. Add targeted unit tests when those constructs gain runtime semantics in Phase 5/6.

## NEXT PHASE: Phase 5 — Interpreter Bridge

- **Goal:** replace the stub `Exec` with a real interpreter that walks the parsed script and dispatches to our command registry (yet-to-land in Phase 8) — backed by `mvdan/sh`'s `interp.Runner` for expansion, redirection, control flow. Per SPEC §5.1, we **do not** translate our AST to mvdan/sh's AST and run interp from there. The strategy is: parse via `parser.Parse` (which already runs `mvdan/sh/syntax`), then run `interp.Runner.Run(ctx, script.Origin)` with extensive `RunnerOption` customization.
- **Read:** SPEC.md §5 in full (§5.1–§5.7) + the Phase 3 handoff's mvdan/sh quirks block (`interp.Dir()` host-stat trap and `interp.HandlerCtx()` panic — both already worked around in `bash.go` and the Phase 5 bridge must preserve them).
- **Deliver:**
  - `interp/runner.go` — `BuildRunner(ctx, *Bash, ExecOptions) (*interp.Runner, error)`. Wires Env, Cwd (via `runner.Dir = cwd`, NOT `interp.Dir(...)`), Stdin/Stdout/Stderr (with the existing MaxOutputSize ringbuf wrapper), VFS handlers (already in `bash.go`), and the command registry's `ExecHandler`.
  - `interp/runner.go` (`commandExecHandler`) — dispatch to the (Phase 8) command registry; for Phase 5 a stub that calls `interp.DefaultExecHandler` is fine until the registry exists, but the wiring point needs to be in place.
  - Cancellation semantics: `ctx` is the single source of truth; no AbortSignal port.
  - Env mutation: per SPEC §5.6, the script's env mutations stay inside the run and do not leak back to `Bash.Env` unless explicitly opted-in.
  - **Public surface to keep stable:** existing `bash.Exec(ctx, script, opts) (BashExecResult, error)` signature must not change — only its internals.
- **Acceptance (SPEC §5.7):** the Phase 1 smoke test `echo Hello > greeting.txt; cat greeting.txt` returns `"Hello\n"` and exit 0 against the VFS (not the host disk). `set -e` propagates. `cd /tmp && pwd` returns `/tmp` from the VFS. Stdin can be piped in via `ExecOptions.Stdin`. Phase 2's limit tests + Phase 3's FS tests + Phase 4's parser tests all continue to pass.
- **Don't:**
  - Implement built-in commands (Phase 8/10).
  - Implement network (Phase 9).
  - Reach for `interp.Dir(path)` as a `RunnerOption` — see the Phase 3 handoff quirk; set `runner.Dir = cwd` directly after `interp.New()`.
  - Drop the `handlerDir` recover-guarded HandlerCtx extraction in the VFS handlers — `interp.HandlerCtx(ctx)` still panics if HandlerContext isn't in ctx.
